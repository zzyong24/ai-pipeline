# Prompt 模板：修复 Bug

## 适用场景

项目中发现 Bug 需要修复时使用。

---

## 模板

```
{{PROJECT_NAME}} 项目中发现一个 Bug，请帮我修复。

## Bug 描述
- 现象：{观察到的错误行为}
- 期望：{正确的行为应该是什么}
- 复现步骤：{列出复现步骤}
- 发生频率：{偶发/必现/特定条件}

## 相关信息
- 日志/错误栈：{粘贴日志}
- 涉及的业务场景：{说明}

## 执行流程
1. 先读取 `.ai/index.md` 定位相关文件
2. 读取 `.ai/known-issues.md` 检查是否已知问题
3. 输出 Phase 1 清单（含 Bug 根因分析）
4. 先分析根因，不要急于修改
5. 修复后检查影响范围（搜索所有调用方）
6. 过 `.ai/review-checklist.md` 中的相关检查项
7. 判断是否需要更新 `.ai/known-issues.md` 或 `.aies/spec/`

## 重要要求
- 不要只修现象，要找到根因
- 修改后必须列出影响范围（调用方列表）
- 如果修复暴露了 Spec 缺口，必须补充 Spec
```

---

## 示例

```
agent-manage-backend 项目中发现 Bug：

## Bug 描述
- 现象：删除 MCP Server 后，agent_mcps 关联表中仍有残留记录
- 期望：删除 MCP Server 时应级联清理所有引用它的 agent_mcps 记录
- 复现步骤：
  1. 创建一个 MCP Server
  2. 创建 Agent 并绑定该 MCP Server
  3. 删除该 MCP Server
  4. 查看 agent_mcps 表，发现关联记录仍在
- 发生频率：必现

## 相关信息
- 业务场景：管理员在后台删除失效的 MCP 服务时
```

---

## Bug 修复五问（AI 必答）

修复前，AI 必须回答：
1. 根因是什么？（不是现象）
2. 有没有其他地方同样错误？（搜索类似模式）
3. 修复后是否破坏已有功能？（列出调用方）
4. 应该写个测试避免回归吗？
5. 这暴露了 Spec 缺口还是实现 Bug？
